# OxideTerm 变更日志 - 2026年2月

## 2026-02-18: IDE 文件树按需加载 & 路径解析修复 (v0.13.2)

### 🔧 架构改进

#### 1. 文件树彻底切换为按需加载

**背景**：原有的深度预取机制（`listTree` depth=3, maxEntries=5000）在大型项目中会产生不完整的目录缓存，导致"文件明明存在但读不到"的反复出现。

**改动**：
- 移除 `PrefetchCacheContext`、`buildPrefetchCache()` 及相关的深度预取调用
- `loadRoot()` 现在仅获取根目录的直接子项，不再后台调用 `listTree(depth=3)`
- 所有子目录完全由用户展开时按需加载（`agentService.listDir`）

**改动文件**：
- `src/components/ide/IdeTree.tsx` — 移除预取缓存系统，改为纯按需加载

#### 2. AbortController 竞态锁

**背景**：用户快速展开/折叠目录时，旧的 `loadingRef` 机制无法取消已发起的请求，可能导致陈旧数据覆盖新数据。

**改动**：
- 新增 `FetchLockContext` — 共享 `Map<path, AbortController>` 传递给所有 TreeNode
- 新请求自动取消同一目录的旧请求（`abort()`），响应到达后检查 `signal.aborted`
- 根目录切换时批量取消所有进行中的请求

**改动文件**：
- `src/components/ide/IdeTree.tsx` — 新增 `FetchLockContext`、`useFetchLock()`

#### 3. Rust Agent 逐目录 truncated 标记

**改动**：
- `FileEntry` 新增 `truncated: bool` 字段，标记该目录的子项列表是否因预算耗尽而不完整
- `list_tree_recursive()` 在全局 `count >= max_entries` 中断循环时设置标记
- 三端同步更新：agent `protocol.rs`、main app `protocol.rs`（`#[serde(default)]` 向后兼容）、TypeScript `types/index.ts`

**改动文件**：
- `agent/src/protocol.rs` — `FileEntry` 添加 `truncated` 字段
- `agent/src/fs_ops.rs` — `list_tree_recursive()` 传播截断标记
- `src-tauri/src/agent/protocol.rs` — 镜像类型同步
- `src/types/index.ts` — TypeScript 类型同步

### 🐛 Bug 修复

#### 4. IDE 根目录使用未规范化路径（关键 Bug）

**现象**：用户在远程文件夹对话框中输入 `~/project` 或切换根目录后，文件树无法读取任何文件。

**根因**：`node_ide_open_project` 将用户输入的原始路径直接作为 `root_path` 返回（可能含 `~`、相对路径、尾斜杠），后续 Agent 的 `Path::new("~/project")` 在 Linux 内核层面会被解析为当前目录下名为 `~` 的子目录，导致所有 `fs::read_dir` 静默失败返回 NotFound。

**修复**：
- 使用 `sftp.stat()` 返回的 `info.path`（经 SFTP `canonicalize` 规范化的绝对路径），替代原始用户输入
- 对 Windows OpenSSH 返回的反斜杠路径做 `replace('\\', "/")` 归一化

**改动文件**：
- `src-tauri/src/commands/node_sftp.rs` — `node_ide_open_project()` 使用 canonicalized 路径
- `src-tauri/src/commands/ide.rs` — `ide_open_project()` 同步修复

#### 5. Agent 不做波浪号（`~`）展开（关键 Bug）

**现象**：Agent 收到含 `~` 的路径时，所有文件操作静默失败。

**根因**：Agent 的所有操作直接使用 `Path::new(&params.path)`，Linux 内核不会展开 `~`（只有 Shell 会）。

**修复**：
- 新增 `resolve_path()` 函数，使用 `$HOME` 环境变量展开 `~` 和 `~/...`
- 应用于所有 12 个文件系统操作及 3 个符号操作

**改动文件**：
- `agent/src/fs_ops.rs` — 新增 `resolve_path()`，全部操作入口替换
- `agent/src/main.rs` — `symbols/*` 方法同步修复

#### 6. `is_absolute_remote_path` 不识别 Windows 驱动器号

**现象**：连接 Windows SSH 服务器时，SFTP `canonicalize` 返回 `C:/Users/...` 格式路径，被错误判断为相对路径。

**修复**：增加 `C:/` / `C:\` 驱动器号检测。

**改动文件**：
- `src-tauri/src/sftp/path_utils.rs` — `is_absolute_remote_path()` 增加 Windows 路径支持

#### 7. DialogTitle 无障碍警告

**改动**：`IdeAgentOptInDialog` 添加 `<DialogTitle className="sr-only">` 消除 Radix UI 屏幕阅读器警告。

**改动文件**：
- `src/components/ide/dialogs/IdeAgentOptInDialog.tsx`

## 2026-02-16: 连接池空闲断连通知 & WsBridge 超时优化 (v1.11.3)

### 🐛 Bug 修复

#### 1. 空闲超时断连未通知前端（关键 Bug）

**现象**：终端放置不动约 1 小时后，前端 UI 仍显示"已连接"，但后端连接池中连接已被移除。

**根因**：`start_idle_timer` 在 30 分钟超时后断开连接时，只通过 `node:state`（Oxide-Next Phase 2）事件发送状态变更，**完全没有发 `connection_status_changed` 事件**。前端的 `useConnectionEvents` hook 监听的是 `connection_status_changed`，因此永远收不到断开通知。

**修复**：在 idle timer 的 spawn 闭包内、断开连接后，补发 `connection_status_changed` 事件（status: `"disconnected"`），确保前端能正确更新 UI、关闭 tabs、清理 profiler 订阅。

**改动文件**：
- `src-tauri/src/ssh/connection_registry.rs` — `start_idle_timer()` 添加 `connection_status_changed` 事件发射

#### 2. WsBridge 本地心跳超时过短

**现象**：切换到其他 App 或 macOS App Nap 约 1.5 分钟后，WsBridge 就判定前端已死，断开本地 WebSocket。

**根因**：`HEARTBEAT_TIMEOUT_SECS` 设为 90 秒。macOS 会对后台 webview 执行 App Nap，暂停 JavaScript 执行，导致前端无法回复心跳 ping。而这个 WebSocket 是 localhost 连接，不经过网络，真实连接活性已由 SSH heartbeat（15 秒间隔）保障。

**修复**：将 `HEARTBEAT_TIMEOUT_SECS` 从 90 秒提升到 300 秒（5 分钟），覆盖绝大多数 App Nap 场景。

**改动文件**：
- `src-tauri/src/bridge/server.rs` — `HEARTBEAT_TIMEOUT_SECS: 90 → 300`

#### 3. `set_keep_alive(false)` 不启动空闲计时器

**现象**：连接设为"永不断开"后，切换回"标准"模式（30 分钟超时），空闲连接不会被自动清理。

**根因**：`set_keep_alive()` 只处理了 `true → 取消计时器`，没处理 `false → 启动计时器`。当连接已是 Idle 状态且 ref_count=0 时，切换到 `false` 后没有启动 idle timer。

**修复**：在 `set_keep_alive()` 中，当 `keep_alive=false` 且连接处于 Idle 且 ref_count=0 时，立即启动空闲计时器。

**改动文件**：
- `src-tauri/src/ssh/connection_registry.rs` — `set_keep_alive()` 添加 false 分支处理

### 📊 受影响常量

| 常量 | 旧值 | 新值 | 位置 |
|------|------|------|------|
| `HEARTBEAT_TIMEOUT_SECS` (WsBridge) | 90s | 300s | `bridge/server.rs` |

## 2026-02-15: SSH 智能重连 & Grace Period (v1.11.2)

### ✨ 新特性

#### 1. SSH 智能重连感知

提升 SSH 断线检测的响应速度，从被动心跳检测（15-30 秒延迟）缩短至 **~2 秒主动探测**：

**前端（TypeScript）**：
- **`visibilitychange` 监听**：检测合盖/休眠恢复，若隐藏时间 > 5 秒自动触发连接探测
- **`online` 事件增强**：网络恢复后 2 秒去抖触发主动探测，而非被动等待心跳超时
- **最小探测间隔**：`MIN_PROBE_INTERVAL_MS = 10000`，防止频繁探测风暴
- **结构化日志**：通过 `slog` 详细记录探测触发原因（wake/online/periodic）

**后端（Rust）**：
- **`probe_active_connections()`**：遍历所有 `Active` 连接发送 SSH keepalive ping，失败则标记 `link_down`
- **`probe_single_connection()`**：针对单条连接的精细探测，支持 `LinkDown` 状态恢复
- **智能重试**：Timeout 结果自动延迟 1.5 秒重试一次（IoError 则直接判定断线）
- **`probe_connections` & `probe_single_connection`**：两个新 Tauri IPC 命令

**改动文件**：
- `src/hooks/useNetworkStatus.ts` — 完全重写（37 行 → 165 行），增加 `visibilitychange` + 主动探测
- `src-tauri/src/ssh/connection_registry.rs` — 新增 `probe_active_connections()` + `probe_single_connection()` 方法
- `src-tauri/src/commands/network.rs` — 新增 2 个 IPC 命令
- `src-tauri/src/lib.rs` — 注册新命令到 `generate_handler!`
- `src/lib/api.ts` — 新增 `probeConnections()` + `probeSingleConnection()` API 包装

#### 2. Grace Period — 优雅恢复机制

引入 **30 秒宽限期**，在执行破坏性重连（销毁旧 SSH 会话）之前尝试恢复现有连接。这对正在使用 TUI 应用（yazi、vim、htop 等）的场景至关重要：

**设计理念**：
- 旧方案「焦土模式」：断线 → 立即销毁旧会话 → 建立新连接。TUI 应用的 PTY 被杀死，所有状态丢失。
- 新方案「宽限恢复」：断线 → **30 秒内每 3 秒探测旧连接** → 若恢复则保留整个会话 → 仅在超时后才执行破坏性重连。

**Orchestrator 管道更新**：
```
snapshot → grace-period (NEW) → ssh-connect → await-terminal → restore-forwards → resume-transfers → restore-ide → verify → done
```

**实现细节**：
- **`GRACE_PERIOD_MS = 30000`**：最大等待 30 秒
- **`GRACE_PROBE_INTERVAL_MS = 3000`**：每 3 秒通过 `api.probeSingleConnection()` 发送 SSH keepalive
- **恢复成功**：返回 `"alive"` → 清除 `link_down` 状态 → 恢复子节点 → 跳过破坏性重连 → 显示 "连接已恢复 — 会话完整保留" Toast
- **恢复失败**：30 秒超时 → 继续执行原有的 `ssh-connect` 阶段（兼容旧逻辑）

**新增常量**：
| 常量 | 值 | 说明 |
|------|-----|------|
| `GRACE_PERIOD_MS` | 30,000 | 宽限期最大等待时间 |
| `GRACE_PROBE_INTERVAL_MS` | 3,000 | 宽限期内探测间隔 |

**改动文件**：
- `src/store/reconnectOrchestratorStore.ts` — 新增 `grace-period` 阶段类型、`phaseGracePeriod()` 实现、`oldConnectionIds` 快照字段、管道插入
- `src/lib/api.ts` — 新增 `probeSingleConnection()` 包装函数

### 🌍 国际化

- 11 种语言新增 2 个键：`reconnect.recovered`（恢复成功提示）+ `reconnect.phase.grace_period`（宽限期阶段名）
- 涉及语言：EN、ZH-CN、ZH-TW、JA、FR、DE、ES、IT、KO、PT-BR、VI

### 🎨 代码质量

- 清理 3 处 Rust 未使用导入警告：
  - `src-tauri/src/ssh/known_hosts.rs` — 移除未使用的 `tempfile::tempdir`
  - `src-tauri/src/ssh/handle_owner.rs` — 移除空测试模块中的 `use super::*`
  - `src-tauri/src/state/session.rs` — 移除未使用的 `AuthMethod` 和 `PathBuf`

## 2026-02-14: 终端会话录制与回放 (v1.11.0)

#### 1. 终端会话录制 (Asciicast v2)

实现完整的终端会话录制与回放功能，支持 asciicast v2 格式（asciinema 标准）：

**后端（Rust）**：
- **`RecordingSession` 结构**：内存中保存 header（cols/rows/env）+ events 数组，支持 v2 序列化
- **`start_recording()`**：验证 sessionId/connectionId，检查 Recording 互斥锁，记录终端尺寸 + 时间戳

- **`stop_recording()`**：返回完整 JSON 内容（NDJSON 格式），自动释放互斥锁
- **`validate_cast_content()`**：解析安全校验（header 必须包含 version/width/height，events 格式校验）
- **并发安全**：DashMap 管理多会话录制，Mutex 保护 RecordingSession 状态

**前端（TypeScript）**：
- **录制状态管理**：`recordingStore` 管理录制中的会话、元数据（elapsed/size），ticker 自动更新

- **保存对话框**：停止录制后自动弹出保存对话框（默认文件名：`{sessionName}_YYYYMMDD_HHMMSS.cast`）
- **快捷键**：`⌘⇧R` / `Ctrl+Shift+R` 全局快捷键启动录制

**安全设计**：
- 录制数据仅存在内存，停止时清除
- v2 格式验证防止恶意构造的 cast 文件
- 录制/回放互斥（同一会话不能同时进行）

## 2026-02-08: 远程环境检测 & AI 上下文增强 (v1.6.1)

### ✨ 新特性

#### SSH 远程环境自动检测

**问题**：AI 内联助手（Inline Chat）和侧边栏聊天仅注入本机 OS 信息（`navigator.platform`），无法为用户提供针对远程服务器 OS/架构/Shell 的准确建议。

**解决**：SSH 连接建立后自动异步检测远程环境，结果注入 AI Prompt：

**检测流程**（双阶段）：
1. **Phase A**（判断平台）：运行 `echo $PSModulePath` —— 非空 = Windows，空 = Unix
2. **Phase B**（采集信息）：
   - **Unix**：`uname -s && uname -r && uname -m && echo "$SHELL"`
   - **Windows**：`systeminfo | findstr /C:"OS Name" /C:"OS Version" /C:"System Type" && echo %COMSPEC%`
3. **MINGW/MSYS/Cygwin 检测**：`uname -s` 输出 `MINGW64_NT-*` / `MSYS_NT-*` / `CYGWIN_NT-*` → `osType = "Windows (MINGW64)"` 等

**RemoteEnvInfo 结构**：
```rust
pub struct RemoteEnvInfo {
    os_type: String,        // "Linux", "Darwin", "Windows", "Windows (MINGW64)", ...
    os_version: String,     // "5.15.0-generic", "10.0.19045", ...  
    kernel: String,         // (Unix only) kernel version
    arch: String,           // "x86_64", "aarch64", "AMD64", ...
    shell: String,          // "/bin/zsh", "cmd.exe", ...
    detected_at: u64,       // Unix timestamp (ms)
}
```

**Tri-State 处理**（异步竞态）：
- `undefined` → 检测进行中，Prompt 显示 `[detecting...]`
- `null` → 检测失败，使用回退提示
- `RemoteEnvInfo` → 正常显示完整环境信息

## 2026-02-07: AI API Key 安全存储升级 (v1.6.0)

### 🔒 安全升级

#### AI API Key 存储迁移至 OS Keychain

**问题**：v1.5.x 及之前版本的 AI API Key 使用 XOR 混淆文件（`ai_keys/*.vault`）存储，安全性等同于明文。XOR 密钥由可预测的机器指纹（`hostname + username`）派生，无密码学保护。

**解决**：将 AI API Key 存储统一迁移至操作系统原生安全存储：
- **macOS**: Keychain Services（`com.oxideterm.ai` 服务）
- **Windows**: Credential Manager
- **Linux**: Secret Service（libsecret / gnome-keyring）
- 与 SSH 密码享有同等 OS 级别加密保护

**改动文件**：
- `src-tauri/src/commands/config.rs`：5 个 `*_ai_provider_*` 命令从 `AiProviderVault` 改为 `Keychain` 调用
- `src-tauri/src/config/vault.rs`：标记为 DEPRECATED，仅保留供迁移读取
- `src-tauri/src/config/mod.rs`：更新模块文档

**迁移机制**：
- **懒迁移**：首次读取 provider key 时自动检测旧 vault 文件 → 解密 → 存入 keychain → 删除 vault 文件
- **零用户干预**：用户无需手动操作，升级后首次使用 AI 时自动完成
- **兼容性**：`has_ai_provider_api_key` 同时检查 keychain 和遗留 vault 文件

**前端**：零改动（Tauri 命令签名不变）

## 2026-02-05: 端口转发内存安全增强 (v1.4.3)

- 旧架构中，读写任务竞争同一个 Mutex 锁
- 持锁调用 `.await` 可能导致其他任务长时间等待
- 某些边缘情况下存在死锁风险

**新架构**
```
┌─────────────────────────────────────────────────────────────┐
│  handle_forward_connection                                  │
│  ┌─────────────┐   mpsc    ┌─────────────────────────────┐  │
│  │ local_reader├──────────►│   ssh_io                    │  │
│  │ (无锁读)    │           │   (单独持有 Channel,        │  │
│  └─────────────┘           │    无需 Mutex)              │  │
│  ┌─────────────┐   mpsc    │                             │  │
│  │ local_writer│◄──────────┤                             │  │
│  │ (无锁写)    │           └─────────────────────────────┘  │
│  └─────────────┘                                            │
│  shutdown_rx ← 来自主任务的 broadcast 信号                  │
└─────────────────────────────────────────────────────────────┘
```

**关键改进**
| 改进项 | 说明 |
|--------|------|
| **Channel 单所有权** | SSH Channel 由 `ssh_io` 任务独占持有，无需 Mutex |
| **mpsc 解耦** | 读写任务通过 `mpsc::channel` 与 SSH I/O 任务通信 |
| **信号传播** | 子连接任务订阅 `shutdown_rx`，SSH 断开时主动退出 |
| **全链路超时** | 所有 I/O 操作都有 `IDLE_TIMEOUT` (300s) 保护 |

#### 子任务 Shutdown 信号传播

修复了子连接任务（每个 accept 的 TCP 连接）未监听 SSH 断开信号的问题：

```rust
// 主任务创建 broadcast 通道
let (child_shutdown_tx, _) = broadcast::channel::<()>(16);

// 每个子任务订阅 shutdown 信号
let mut child_shutdown_rx = child_shutdown_tx.subscribe();

// 子任务内部监听
tokio::select! {
    _ = shutdown_rx.recv() => {
        debug!("Received shutdown signal");
        break;
    }
    // ... 其他 I/O 操作
}
```

#### remote.rs 超时保护

为 `bridge_forwarded_connection` 添加了缺失的 `REMOTE_FORWARD_IDLE_TIMEOUT` (300s)：

```rust
// 所有 I/O 都有超时保护
match tokio::time::timeout(REMOTE_FORWARD_IDLE_TIMEOUT, channel.wait()).await {
    Err(_) => {
        debug!("idle timeout, closing connection");
        break;
    }
    // ...
}
```

## 2026-02-04: 连接架构修复 (v1.4.0)

### 🐛 Bug 修复

#### Store 间同步缺失导致 `connectionState=undefined` 死锁

**问题现象**：
- SSH 重连成功后，SFTPView 和 TransferQueue 卡在 "Waiting for connection"
- 控制台显示 `connectionState=undefined, isConnectionReady=false`
- StateDrift 日志显示 `Auto-fixed 1 drift(s)` 但 UI 无反应

**根本原因**：
`sessionTreeStore.connectNodeInternal()` 和 `syncDrift()` 只更新 `rawNodes`，
不同步 `appStore.connections`，而 SFTPView/TransferQueue 依赖后者判断连接状态。

```
旧数据流（断裂）:
  [Backend] SSH Connected
    → [sessionTreeStore] rawNodes 更新 ✅
    → [appStore] connections Map 未更新 ❌
    → [SFTPView] connectionState = undefined → 卡死 💀
```

**解决方案**：
在以下位置添加 `await useAppStore.getState().refreshConnections()`：

| 文件 | 位置 | 触发时机 |
|------|------|----------|
| `sessionTreeStore.ts` | `connectNodeInternal()` | SSH 连接建立成功后 |
| `sessionTreeStore.ts` | `syncDrift()` | StateDrift 自动修复后 |

```
新数据流（完整）:
  [Backend] SSH Connected
    → [sessionTreeStore] rawNodes 更新
    → [refreshConnections()] appStore.connections 同步
    → [SFTPView] connectionState = 'active' → 正常初始化 ✅
```

**影响范围**：
- SFTPView
- TransferQueue
- ForwardsView
- IdeWorkspace
